# Story 2.6.0: Foundation — Infrastructure Hardening

## Status: Ready

## Executor
executor: "@dev"
quality_gate: "@qa"

## Story
**As a** operador do ecossistema Vialum,
**I want** que a infraestrutura base seja resiliente, sem pontos únicos de falha e com configurações seguras,
**so that** os serviços não percam dados, não falhem silenciosamente e estejam preparados para escalar.

## Acceptance Criteria
1. [ ] Redis com 2GB de memória + persistence AOF (não perde jobs se reiniciar)
2. [ ] Redis eviction policy: noeviction (rejeita novos jobs se cheio, não descarta)
3. [ ] Alerta quando Redis > 80% capacidade (log warn periódico)
4. [ ] WEBHOOK_SECRET obrigatório no Switch (falha no startup se vazio)
5. [ ] DB connection pool explícito: 30 no Chat, 15 no Hub, 15 no Media, 15 no Switch
6. [ ] Media Service com 512MB de memória
7. [ ] Switch com 512MB de memória
8. [ ] Remover dependência circular: Hub NÃO lê banco do Chat (VIALUM_DATABASE_URL removido)
9. [ ] Health check endpoint em todos os serviços (GET /health retorna 200 com status do DB + Redis)
10. [ ] Docker Compose: health checks + depends_on com condition: service_healthy
11. [ ] Contact creation atômico: upsert com ON CONFLICT (account_id, phone) no webhook worker

## Dev Notes
- O Redis compartilhado é aceitável nesta fase. Separar Redis por serviço é otimização futura.
- Connection pool de 30 pro Chat é pra suportar: 10 webhook workers + 5 media workers + 10 message-send + API requests.
- A remoção da dep circular precisa ser feita COM CUIDADO. Primeiro mapear, depois criar alternativas, depois remover.
- O upsert atômico no contact usa a constraint unique pra garantir que o banco rejeita duplicatas, não a aplicação.
- AOF persistence do Redis: `appendonly yes` no command do docker-compose.
