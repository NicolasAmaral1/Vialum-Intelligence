# Story 2.6.0: Foundation — Infrastructure Hardening

## Status: Ready

## Executor
executor: "@dev"
quality_gate: "@qa"

## Story
**As a** operador do ecossistema Vialum,
**I want** que a infraestrutura base seja resiliente, sem pontos únicos de falha e com configurações seguras,
**so that** os serviços não percam dados, não falhem silenciosamente e estejam preparados para escalar.

## Acceptance Criteria
1. [ ] Redis com 2GB de memória + persistence AOF (não perde jobs se reiniciar)
2. [ ] Redis eviction policy: noeviction (não descarta jobs — retorna erro se cheio)
3. [ ] WEBHOOK_SECRET obrigatório no Switch (falha se vazio)
4. [ ] DB connection pool explícito: 20 connections no Chat, 10 no Hub, 10 no Media, 10 no Switch
5. [ ] Media Service com 512MB de memória (suportar arquivos grandes)
6. [ ] Switch com 512MB de memória (suportar ML inference)
7. [ ] Remover dependência circular: CRM Hub NÃO lê banco do Chat (VIALUM_DATABASE_URL removido)
8. [ ] Health check endpoint em todos os serviços (GET /health retorna 200)
9. [ ] Docker Compose: health checks + depends_on com condition: service_healthy
10. [ ] Contact creation transacional no webhook worker (prevenir duplicatas)

## Dev Notes
- O Redis compartilhado é aceitável nesta fase. Separar Redis por serviço é otimização futura.
- A remoção da dependência circular (Hub → Chat DB) pode impactar a sidebar do CRM. Verificar se alguma feature do Hub depende de ler o banco do Chat diretamente.
- O connection_limit do Prisma é passado na connection string: `?connection_limit=20`
- AOF persistence do Redis: `appendonly yes` no command do docker-compose
