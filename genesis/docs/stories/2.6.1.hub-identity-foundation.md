# Story 2.6.1: Vialum Hub — Identity Foundation

## Status: Ready

## Executor
executor: "@dev"
quality_gate: "@qa"

## Story
**As a** desenvolvedor do ecossistema Vialum,
**I want** que o Hub seja a fonte única de verdade para identidade de contatos,
**so that** qualquer serviço consiga resolver um telefone/email/ID externo para um contato universal com todos os dados de CRM, tasks e documentos.

## Acceptance Criteria
1. [ ] Sync de providers movido pra BullMQ — 2 filas: `provider-sync-priority` (on-demand) e `provider-sync-background` (stale refresh)
2. [ ] `resolve` retorna dados cacheados imediatamente + enfileira sync na fila background se stale
3. [ ] Staleness per-provider — campo `lastSyncAt` na tabela `provider_configs` per-tenant
4. [ ] `syncContactNameFromCRM` no Chat escreve em `crmName` (não sobrescreve `name`)
5. [ ] Chat Contact ganha campo `hubContactId` (nullable UUID)
6. [ ] Webhook worker do Chat: após criar contato, enfileira job `hub-ensure` com retry (não fire-and-forget)
7. [ ] Hub `POST /contacts/ensure`: idempotente, recebe phone + sourceId + source, retorna hubContactId
8. [ ] HubContact usa `externalSourceId` + `externalSource` (não `chatContactId` — agnóstico)
9. [ ] Hub não gera UUID fantasma em `externalSourceId` — usa o ID real recebido do serviço de origem
10. [ ] Alias types extensíveis — registry de tipos conhecidos (phone, email, cpf, cnpj, pipedrive_person, clickup_member, etc.) + custom com prefixo `custom:`
11. [ ] Sync salva external IDs como aliases (ex: `pipedrive_person:12345`)
12. [ ] Resolução reversa funciona: dado `pipedrive_person:12345`, encontra o HubContact
13. [ ] Per-tenant rate limit no sync via Redis-based semaphore (BullMQ rate limiter), não in-memory Map

## Dev Notes
- O Hub hoje não tem Redis. Precisa adicionar ao docker-compose e env.
- O `ensure` é diferente do `resolve`: ele NÃO dispara sync de providers. Só cria/encontra contato e retorna ID.
- `externalSourceId` + `externalSource` permite que qualquer serviço (Chat, Portal, outro) crie contatos no Hub.
- A fila `provider-sync-priority` é pra quando operador clica "refresh" ou IA precisa de dados frescos.
- A fila `provider-sync-background` é pra staleness refresh — baixa prioridade, pode esperar.
- O registry de alias types é uma constante no código, não no banco. Novos tipos = PR + deploy.
- BullMQ rate limiter é nativo: `new Worker('queue', handler, { limiter: { max: 2, duration: 1000, groupKey: 'accountId' } })`.
