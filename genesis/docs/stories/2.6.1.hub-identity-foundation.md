## Status: Ready

## Executor
executor: "@dev"

## Story
**As a** desenvolvedor do ecossistema Vialum,
**I want** que o Hub seja a fonte única de verdade para identidade de contatos,
**so that** qualquer serviço consiga resolver um telefone/email/ID externo para um contato universal com todos os dados de CRM, tasks e documentos.

## Acceptance Criteria
1. [ ] Sync de providers movido pra BullMQ queue (não síncrono dentro do resolve)
2. [ ] `resolve` retorna dados cacheados imediatamente + enfileira sync se stale
3. [ ] Staleness per-provider (não global) — só re-sync provider que está stale
4. [ ] `syncContactNameFromCRM` no Chat escreve em `crmName` (não sobrescreve `name`)
5. [ ] Chat Contact ganha campo `hubContactId` (nullable UUID)
6. [ ] Webhook worker do Chat: após criar contato, chama Hub `POST /contacts/ensure` e salva hubContactId
7. [ ] Hub `POST /contacts/ensure`: idempotente, recebe phone + chatContactId, retorna hubContactId
8. [ ] Hub não gera `vialumContactId` fantasma — usa o chatContactId recebido
9. [ ] Alias types extensíveis — qualquer string como type (não só phone/email/cpf)
10. [ ] Sync salva external IDs como aliases (ex: `pipedrive_person:12345`)
11. [ ] Resolução reversa funciona: dado `pipedrive_person:12345`, encontra o HubContact
12. [ ] Per-tenant rate limit no sync (semáforo — max 2 provider calls concorrentes por tenant)

## Dev Notes
- O Hub hoje não tem Redis. Precisa adicionar ao docker-compose e env.
- O `ensure` endpoint é diferente do `lookup`: ele NÃO dispara sync. Só cria/encontra contato e retorna ID.
- O sync é disparado separadamente via queue, quando o contato é acessado via `resolve` e está stale.
- A resolução reversa (ex: buscar por `pipedrive_person:12345`) usa a mesma tabela de aliases, só com type diferente.
- Per-tenant rate limit: usar Map<accountId, Semaphore> no worker. Max 2 concurrent provider calls por tenant.
- O `chatContactId` no Hub substitui o `vialumContactId` — é o ID real do contato no Chat, não um UUID fantasma.

## Dependências
