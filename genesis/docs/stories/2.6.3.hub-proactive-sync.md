# Story 2.6.3: Vialum Hub — Proactive Sync + External ID Aliases

## Status: Draft

## Executor
executor: "@dev"
quality_gate: "@qa"

## Story
**As a** operador do Vialum,
**I want** que o Hub importe contatos dos providers periodicamente e salve IDs externos como aliases,
**so that** um contato no ClickUp seja automaticamente vinculado ao mesmo contato no Pipedrive
e no WhatsApp, sem intervenção manual.

## Acceptance Criteria
1. [ ] Job periódico (cron) que importa contatos/deals ativos de cada provider por tenant
2. [ ] Intervalo configurável per-tenant (campo `syncIntervalMinutes` em provider_configs, default 240 = 4h)
3. [ ] Stagger per-tenant: distribui tenants uniformemente no intervalo do cron
4. [ ] Rate limit per-provider per-tenant respeitado (usa BullMQ limiter)
5. [ ] Após sync, IDs externos salvos como aliases no registry de tipos:
   - `pipedrive_person:12345`
   - `pipedrive_deal:67890`
   - `clickup_member:abc`
   - `rdstation_contact:def`
6. [ ] Resolução reversa: `POST /identity/resolve { externalId: "12345", provider: "pipedrive" }` encontra o contato
7. [ ] Match automático entre providers SOMENTE por email (exato) — nome é apenas sugestão pro operador
8. [ ] Dedup: phone normalizado antes de comparar (usa normalizePhone existente)
9. [ ] Se contato já existe (por phone ou email), não cria duplicata — linka alias novo ao existente
10. [ ] Endpoint GET /admin/sync-status por tenant (último sync, contagem, erros)

## Tasks

### Hub — Proactive Sync Job
- [ ] Criar worker cron `proactive-sync.worker.ts` — roda a cada 30min, verifica quais tenants precisam sync
- [ ] Per-tenant: checar `provider_configs.lastSyncAt` + `syncIntervalMinutes` pra decidir se precisa sync
- [ ] Stagger: distribuir tenants no intervalo (hash do accountId % intervalo)
- [ ] Pra cada tenant+provider: enfileirar na fila `provider-sync-background`

### Hub — External ID Aliases
- [ ] Modificar sync.ts: após sync de um provider, pra cada resource retornado, salvar alias:
  - type: `{provider}_{resourceType}` (ex: `pipedrive_person`)
  - value: externalId
- [ ] Dedup de aliases: upsert com ON CONFLICT

### Hub — Cross-Provider Linking
- [ ] Ao importar contato de provider X com email, buscar alias `email:{email}` em contatos existentes
- [ ] Se encontrar: linkar resources do provider X ao contato existente (não criar novo)
- [ ] Se não encontrar por email: criar novo HubContact
- [ ] NUNCA linkar automaticamente por nome — risco de falso positivo multi-tenant

### Hub — Admin Endpoint
- [ ] GET /admin/sync-status — retorna per-tenant: { lastSyncAt, contactCount, errorCount, providers: [{name, lastSyncAt, status}] }
- [ ] Proteger com adminGuard

### Hub — Provider Config
- [ ] Adicionar campo `syncIntervalMinutes` (int, default 240) na tabela provider_configs
- [ ] Migration: ADD COLUMN sync_interval_minutes INT DEFAULT 240

## Dev Notes
- O proactive sync NÃO substitui o on-demand sync (que roda quando contato é acessado via resolve).
- O stagger usa hash simples: `hash(accountId) % totalSlots` pra distribuir tenants uniformemente.
- O cron roda a cada 30min mas só enfileira tenants cujo `lastSyncAt + syncIntervalMinutes` já passou.
- ClickUp não tem busca por phone — import é por listagem de members + match por email.
- Se um tenant tem syncIntervalMinutes = 0, desabilita proactive sync pra esse tenant.
- Phone normalization já existe no Hub (`lib/phone.ts`). Usar SEMPRE antes de dedup.
- Dashboard frontend é story separada — aqui só o endpoint de backend.
